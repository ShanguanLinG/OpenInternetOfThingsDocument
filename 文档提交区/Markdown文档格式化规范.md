# Markdown文档格式化规范

## 一、基本要求

1. **统一编码格式**：所有文档统一使用UTF-8编码格式。
2. **标点符号规范**：
    * 必须使用英文标点：`,`, `.`, `:`, `;`, `!`, `?`
    * 必须使用弯引号：`‘`, `’`, `“`, `”`
    * 避免使用全角标点。
3. **内容忠实原则**：严格忠实于原文，不添加额外解释或评论。
4. **简洁原则**：文字简洁，删除冗余说明。
5. **预检查原则**：处理前确认源文件编码格式正确，无乱码字符。
6. **列表使用原则**：统一使用无序列表，避免使用序号列表，以保持格式简洁和一致性。
7. **内容完整原则**：禁止脱离原文档内容进行总结、引申或添加额外章节（如“故障排除”、“注意事项”、“验证步骤”等），仅对现有内容进行格式转换与规范化。
8. **结构化原则**：对原文中具有明显列表或表格特征但未格式化的内容（例如“名称:
   值”对、枚举项等），应优先转换为Markdown表格。原文中的文字描述保持段落或列表形式。

## 二、格式处理规范

### 1. 超链接处理

文档中出现以下表述时使用Markdown链接：

* “参考[文档名称]”
* “查看[文件名]”
* “单击[文档名]”
* “打开[参考文件]”

**链接格式：**

```
[链接文本](链接地址)
```

**要求：**

* 链接文本和链接地址的文件名（含后缀）应完全一致。
* 链接地址使用相对路径或文件名。
* 若原文没有明确链接地址，则使用文件名作为链接。

### 2. 任务步骤格式

**使用场景**：用于描述操作步骤，特别是包含硬件连接的操作。

**格式规范：**

```
## [任务名称]

### 准备工作
*   **[关键步骤1]：[关键说明]**
    详细要求1
    详细要求2
    示例或示意图说明
*   **[关键步骤2]：[关键说明]**
    详细要求1
    详细要求2
    示例或示意图说明
```

### 3. 参数配置格式

**使用场景**：任何需要填写选项或参数的地方。

**格式：**

```
### [配置项名称]
[配置位置说明]

| 参数名   | 参数值/说明   |
|-------|----------|
| [参数1] | [值1/说明1] |
| [参数2] | [值2/说明2] |
```

### 4. 图形化数据格式

**使用场景**：任何用字符表示结构、图表、接口部分的描述。

**识别方法：**

1. **引脚/接口描述**：使用空格分隔的字符序列。
    * 例：`A B C D E F G H I`
    * 例：`DI1 DI2 DI3 DI4 DI5 DI6 DI7 DI8 GND`
2. **图表/图示类**：使用特殊字符表示连接或状态。
    * 例：`* * & & & & & & &`
    * 例：使用 `&` 表示连接，`*` 表示端点。

**转换步骤：**

1. **确定列数**：根据空格分隔的字符序列确定列数。
2. **生成表头**：将序列信息转为表头，或使用通用表头。
3. **提取数据**：每行转换为表格的一行数据。
4. **保留原义**：特殊字符（`&`, `*` 等）原样保留。

**通用表格格式：**

```
| 位1  | 位2  | 位3  | 位4  | 位5  | 位6  | 位7  | 位8  | 位9  |
|------|------|------|------|------|------|------|------|------|
| [字符] | [字符] | [字符] | [字符] | [字符] | [字符] | [字符] | [字符] | [字符] |
```

### 5. 名词术语格式

**使用场景**：解释专业术语或名词。

**格式：**

```
### [术语类别]说明
| 术语/缩写  | 说明    |
|-------|-------|
| [术语1] | [说明1] |
| [术语2] | [说明2] |
```

### 6. 代码块保留规范

**使用场景**：命令行指令、代码片段、配置示例等。

**格式：**

```
# [文档标题]

## [章节标题]

### 代码示例
*   **[代码描述]：[简要说明]**
    ```[语言标识]
    [代码/命令内容]
    ```
```

**要求：**

1. 命令行指令、代码等必须使用代码块包裹。
2. 代码块前应有简要说明。
3. 代码块使用三个反引号（```）作为开始和结束标记。
4. 在代码块开始处标记编程语言标识（如 bash、python）。
5. 代码内容保持原样，不进行格式转换。

### 7. 枚举列表转表格

**使用场景**：原文中出现的具有明显并列结构的多行“名称: 值”、“设备: 参数”等描述，且条目数量适中。

**识别方法：**

* 条目清晰，每行通常包含一个名称或项目及对应的说明或数值。
* 行首可能有序号、项目符号或破折号，也可能仅为纯文本。
* 内容通常用于列举、配置项说明、设备列表等。

**转换格式：**

```
### [相关主题]

| [列名1] | [列名2] | (可选)[列名3] |
|-------|-------|-------------|
| [条目1名称] | [条目1说明/值] | ... |
| [条目2名称] | [条目2说明/值] | ... |
```

**要求：**

* 根据上下文为表格设置合适的标题和列名。
* 列名应准确概括该列的内容属性（如“设备名称”、“默认波特率”、“说明”等）。
* 保持原文的条目顺序和内容。
* 如果原文条目自带简单的层级或分组（如缩进表示子项），可考虑使用嵌套表格或在“说明”列内用列表表示。

**示例（原始文本）：**

```
常见设备默认波特率：
    二氧化碳变送器（485型）：9600
    水浸传感器：9600
    ...
```

**转换后：**

```
### 常见设备默认波特率
| 设备名称 | 默认波特率 |
|-------|----------|
| 二氧化碳变送器（485型） | 9600 |
| 水浸传感器 | 9600 |
| ... | ... |
```

## 三、特殊数据处理

### 1. 字符图形识别与处理

**步骤1：识别图形内容**

* 检查是否包含特殊字符（`&`, `*`, `?`, `$` 等）。
* 检查是否有规律的字符排列。
* 判断是否表示接口部分或状态图。

**步骤2：转换为表格**

* 根据空格分隔确定列数。
* 每行作为表格的一行数据。
* 保持特殊字符原样。

**步骤3：添加说明**

* 将关于状态或布局的说明文字转为表格注释。
* 将接口引脚说明转为表头。

### 2. 参数列表处理

文档中包含多组参数时：

1. 为每组参数添加标题说明。
2. 保持组与组之间的逻辑关系。
3. 使用空行分隔不同组别。

## 四、整体格式合规规则

### 1. 预处理

**在处理任何文档前需检查：**

* 文件编码格式（确认使用UTF-8）。
* 是否存在不可见字符。
* 是否存在乱码问题。

### 2. 格式转换

**遇到非标准格式时：**

1. 识别原始格式的意图。
2. 按照最接近的规范进行转换。
3. 在转换后的内容附近添加格式说明。

### 3. 质量验证

**完成文档后需验证：**

* 所有链接格式正确。
* 所有表格结构完整。
* 所有代码格式正确且可读。
* 整体格式统一。
* 无乱码字符。
* 防止在转换过程中丢失原文的重要信息。
* 文档结构层级清晰合理。

### 4. 问题处理流程

**遇到处理错误时：**

1. 停止当前处理。
2. 检查源文件状态。
3. 尝试以不同编码格式重新读取。
4. 若确实无法解析，则在原文处添加注释说明。

### 5. 格式污染防范

**防止格式杂乱的措施：**

1. 所有命令、代码、配置数据必须使用代码块。
2. 所有参数列表、选项说明使用表格分隔。
3. 所有标题使用正确的层级的Markdown标记。
4. 确保转换结果为纯文本Markdown格式，而非特定编辑器的专有格式。
5. 直接删除转换说明等任何非必要语句，如“以下是根据《Markdown文档格式化规范》转换后的...文件”。

## 五、文档结构规范

### 1. 标题层级

* 一级标题：`#` 文档主标题
* 二级标题：`##` 主要章节
* 三级标题：`###` 子章节或功能模块
* 四级标题：`#` 细节描述（尽量避免使用）

### 2. 段落组织

* 每个主要章节后应有空行。
* 列表项与正文之间应有空行。
* 表格前后应有空行。
* 代码块前后应有空行。
